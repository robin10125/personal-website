Neural Network in C++

I made a neural network framework in C++.

This project is a small neural network library written in C++17. It provides the core building blocks needed to define, train, and run feedforward dense neural networks. A standalone demo program at the repository root trains a multilayer perceptron on MNIST handwritten digits. The library follows the conceptual model popularized by PyTorch. A `Tensor` is the universal data carrier. Operations on tensors record a graph of `Node` objects, and that graph drives reverse mode automatic differentiation. A `Parameter` wraps a `Tensor` with the metadata needed for optimization. Layers wrap tensor operations, and loss functions, an optimizer, a trainer, and a data loader sit above this and orchestrate the training loop. The library has no third party dependencies beyond the C++ standard library. Random numbers come from ``, file parsing uses ``, and storage is managed through `std::shared_ptr` and `std::unique_ptr`. The build is a single `g++` invocation that compiles all `.cpp` files together.

Architecture

The codebase has roughly four conceptual layers: at the bottom sits the **tensor and autograd core** in `tensor.h` and `tensor.cpp`. This layer defines storage, views, and tensor operations like arithmetic, reductions, matrix multiplication, activation primitives, and the autograd graph. Almost every other file in the project consumes this header. On this sits the **parameter layer**. `Parameter` lives in `parameter.h` and `parameter.cpp` and wraps a learnable tensor. Layers, linear `LinearLayer` and `ReLULayer` are thin wrapper objects that own parameters or simply delegate to a tensor activation. Above the layers sits the **training infrastructure**. `Loss` and its subclasses compute scalar loss values and seed the backward pass. `Optimizer` implements the Adam update rule and owns moment buffers per parameter. `Trainer` ties the forward pass, backward pass, optimizer, loss, and data loader into a single training loop. `Serialization` writes parameter tensors to disk and reads them back. `DataLoader` turns CSV or binary files into batches of tensors. Finally we have a **demo application**. `mnist_demo.cpp` reads the IDX format MNIST files, exports them to CSV, builds a three layer MLP, trains it for several epochs, and reports test accuracy.

The Tensor Type

The `Tensor` class is the main data structure for this framework. I'll start with explaning supporting functions, and finishb with the matrix math `Tensor` implements. It holds five pieces of view metadata along with a shared storage block. The storage is a `std::shared_ptr`. `Storage` itself owns a `std::unique_ptr` and a size. Multiple `Tensor` views can share the same storage by holding different shapes, strides, and offsets. This is what allows `transpose` and `reshape` to avoid copies. The metadata fields are a `shape_` vector of `int64_t`, a `strides_` vector, an `offset_` measured in elements, and a cached `numel_`. The `numel_` field is the product of the shape and is computed once at construction. Contiguity is not stored. It is checked on demand by comparing the strides against the canonical row major strides for the shape. Construction goes through several named factories. `Tensor::empty` allocates uninitialized storage. `Tensor::zeros` and `Tensor::ones` fill the storage with constants. `Tensor::randn` draws values from a normal distribution from `std::random_device`. `Tensor::from_vector` copies a flat `std::vector` into a fresh storage with the requested shape. `Tensor::scalar` produces a rank zero tensor with a single element. View operations stay cheap. `reshape` returns a new `Tensor` that points at the same storage but uses the new shape's canonical strides. It only succeeds when the source tensor is already contiguous and the element counts agree. `transpose` swaps two entries in both the shape and stride vectors and otherwise copies the view. `contiguous` allocates a fresh tensor and walks the source tensor in row major order, computing each source offset from the strides. `detach` produces a tensor that shares storage but drops its autograd history. Element access goes through `data()` plus the offset. The `data()` method returns `storage_->ptr() + offset_`. A scalar tensor exposes its single value through `item()`, which throws when the tensor has any nonzero rank.

Tensor arithmetic implements the set of operations needed for basic dense neural networks. These functions are `add`, `sub`, `mul`, `div`, `exp`, `log`, `sum`, `mean`, `matmul`, `relu`, and `sigmoid`. The elementwise operations follow a uniform pattern. They check that both inputs share the same shape. They enforce contiguity through `ensure_contiguous_or_throw`. They decide whether the result needs gradient tracking by combining the global `GradMode::enabled()` flag with the `requires_grad` flag of each input. They allocate an output tensor of the same shape. They run a single pass over the data. If gradient tracking is on, they attach an autograd node to the output. The reductions `sum` and `mean` handle two cases. When no axis is supplied, they reduce over every element and produce a scalar. When an axis is supplied, they compute outer and inner stride products around the chosen axis and run a triply nested loop that accumulates into the output. `mean` reuses `sum` and then scales the result by the reciprocal of the reduced dimension. `matmul` is restricted to two dimensional inputs. It checks that the inner dimensions match, allocates an output of shape `{m, n}`, and runs the classical triply nested loop. `relu` is a simple max with zero. `sigmoid` uses the logistic formula. Both operations track gradients through a saved input or saved output. Broadcasting support is partial. The arithmetic operations require matching shapes. The helper `reduce_to_shape` does walk a gradient back to a smaller shape during the backward pass by summing along axes of size one, which lets the autograd nodes for `add` and `sub` cope with shape mismatches that would arise if broadcasting were extended later.

Autograd

Autograd is built around a `Node` interface and a per tensor `grad_fn_` slot. A `Node` knows the parent tensors of an operation and how to propagate gradients to them. Every operation that needs to be differentiable defines a `Node` subclass and attaches an instance to the output tensor through `set_grad_fn`. The base `Node` keeps its parents alive by storing them as `Tensor` values inside a `std::vector` field. This keeps the autograd graph reachable until the user releases all tensors that point into it. The `Tensor` class carries the autograd state in three fields. `requires_grad_` records whether the user wants gradients for the tensor. `is_leaf_` records whether the tensor was created directly or produced by an operation. `grad_slot_` is a `std::shared_ptr` to a `std::shared_ptr`. The double indirection lets `Tensor` copies share a single gradient buffer. When one copy of a tensor accumulates a gradient, every other view that was copied from the same source sees it through the shared slot. Gradient accumulation runs through `accumulate_grad`. It allocates a zero tensor of the parent's shape on the first call, checks that the incoming gradient is contiguous and has a matching shape, and adds the values element by element. The public `add_grad` method forwards to it. The `zero_grad` method resets the slot to a null pointer so that the next backward pass starts from a clean state. The `backward` method runs reverse mode differentiation. The scalar overload constructs a tensor of value one and forwards to the general overload. The general overload first stores the input gradient in the tensor's slot. It then performs a depth first traversal of the graph that begins at the current tensor and walks through `grad_fn_->parents`. The traversal records each visited node and pushes the parent tensors onto a topological order vector. The reversed vector is then iterated, and each node's `backward` method is called with the gradient that has been accumulated in its own slot. Here is a quick overview of node implementation: `AddNode` pushes the upstream gradient to both parents, reducing as needed. `SubNode` does the same but negates the gradient for the second operand. `MulNode` and `DivNode` apply the product rule and the quotient rule, building intermediate tensors with `mul`, `sub`, and `div`. `ExpNode` multiplies the upstream gradient by the exponential of the input. `LogNode` divides the upstream gradient by the input. `MatMulNode` performs the two transposed matrix multiplications that produce gradients for the left and right operands. `ReluNode` zeros out entries where the saved input was non positive. `SigmoidNode` uses the cached output and the identity `sigma * (1 - sigma)`. Gradient tracking can be turned off globally through `GradMode::set_enabled`. A thread local boolean controls the flag. `NoGradGuard` is an RAII helper that flips the flag off on construction and restores it on destruction. The `InferenceMode` class in `inference_mode.h` is a thin wrapper around the same mechanism that is documented as the user facing name for the same RAII pattern.

The Parameter Class

`Parameter` is a small adapter around `Tensor` that captures the metadata needed for optimization. You can think of it as the container that holds tensors for the layers, and handles tensor learnability. It stores a `Tensor` value, a `std::string` name, and a `trainable_` boolean. The default constructor is deleted, so every `Parameter` must be created with explicit data. The factory methods mirror those on `Tensor`. `Parameter::zeros`, `Parameter::ones`, and `Parameter::randn` build a tensor with the requested shape and wrap it in a parameter. Each factory forwards the optional name and trainable flag and ensures the underlying tensor has `requires_grad` set in line with the trainable flag. The class exposes the wrapped tensor through `tensor()` and `data()`. The two accessors are aliases. `grad()` returns the underlying tensor's gradient slot, which lets the optimizer read the most recent gradient. `trainable()` and `set_trainable(bool)` toggle the `requires_grad` flag on the wrapped tensor, which freezes or unfreezes the parameter for optimization. `zero_grad()` clears the gradient slot. `detach_()` drops the parameter's autograd history while keeping its value. The `enforce_invariants_` helper exists to re establish those internal consistency rules whenever state mutates. The class is used in two places. Layers create parameters in their constructors and expose them through a `parameters()` method that returns raw pointers. The trainer collects those pointers and forwards them to the optimizer.

Parts Implemented by Group Members

While I wrote the parts listed above, the following sections were only architected by me, and were implemented by my group members. I will mention them in passing.

Layers

The layer classes are a thin wrapper around tensor operations.

Optimizer

`Optimizer` in `optimizer.h` implements the Adam algorithm. It is constructed with a learning rate, a first moment decay rate beta1, and a second moment decay rate beta2. The defaults match the canonical Adam values of 0.001, 0.9, and 0.999. The optimizer keeps a list of `Parameter*` and a hash map of per parameter state. Each state entry holds a first moment tensor `m` and a second moment tensor `v`.

Loss Functions

`Loss` in `loss.h` is an abstract base class. It defines a `compute` method that takes a prediction and a target tensor and returns a scalar loss tensor, and a `backward` method that produces the gradient of the loss with respect to the prediction. The result of the backward call is stored in an internal slot and exposed through `get_input_grad`. Two concrete losses are provided. `MSELoss` implements mean squared error. Its forward pass computes the elementwise squared difference and reduces by the mean. Its backward pass returns twice the difference divided by the element count. The class caches the prediction values, target values, and shape from the forward pass so that the backward call can run without re reading the source tensors. `CrossEntropyLoss` operates on logits of shape `{N, C}` and integer class indices of shape `{N}`.

Trainer

`Trainer` in `trainer.h` orchestrates the training loop. It is constructed with a forward function, a backward function, a list of parameters, an optimizer, a loss, an epoch count, and an optional target extraction function. The constructor registers all parameters with the optimizer. `train` iterates over a `DataLoader`, calls the forward function with each batch, derives a target tensor from the same batch through the target function, runs the loss forward, runs the loss backward, hands the resulting gradient to the supplied backward function, and asks the optimizer to step and zero the gradients. The trainer accumulates the running mean of the loss into a metrics map under the key `train_loss`. `validate` is the equivalent loop run inside an `InferenceMode` guard. `prepareData` resets the loader and optionally shuffles the data. `checkpoint` and `loadCheckpoint` delegate to the serialization helpers. The remaining methods expose epoch configuration and metric retrieval.

Data Loader

`DataLoader` in `dataloader.h` is responsible for turning files into batches. It supports two file formats. A CSV format with a numeric header line and one sample per line, and a binary format that begins with two `uint64` fields for the sample count and feature count, followed by a flat block of `float32` values.

The class loads every sample into memory at once. Samples are stored as a vector of vectors of floats. A permutation array of indices supports shuffling without moving data. `load_data` reads the file, populates the sample table, and resets the iteration state. `get_next_batch` reads up to `batch_size` indices, copies the corresponding rows into a fresh contiguous tensor of shape `{B, F}`, and advances the cursor. The final batch of an epoch may be smaller than the configured batch size. `epoch_done` reports when the cursor has run off the end. `reset` rewinds the cursor and optionally reshuffles.

Serialization

`Serialization` in `serialization.h` is a static class with two methods. `save` opens a binary output stream, walks the supplied parameter list, and writes each parameter as a header consisting of the rank and the shape followed by the raw float data. `load` reads the same format back and copies the bytes into the matching parameter slots. The format is positional. The parameter order at load time must match the order at save time. The trainer's `checkpoint` and `loadCheckpoint` methods are thin wrappers around these calls.

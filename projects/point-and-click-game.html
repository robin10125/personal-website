Point and Click Adventure Game

This project is a Java-based point-and-click adventure game built as a university group project to demonstrate object-oriented programming and software engineering practices.

The game, themed around the *Grand Boqueria Market*, casts the player as a visitor competing for the "Golden Apple Trophy" by collecting at least 40 tickets within a 75-turn limit. The player navigates between themed market locations (the Entrance Gate, Main Square, Fresh Produce Row, Dairy & Livestock Corner, Picnic Grove, Candy Land, Arts & Crafts Lane, Stage Plaza, Storage Barn, and Tools & Hardware Stand) by clicking on directional connections, interacting with characters, picking up objects, using items on objects, giving items to vendors, and solving short item-based puzzles. Each location is rendered with a hand-illustrated PNG backdrop, and gameplay is driven by a graphical Swing-based interface rather than text input.

The deliverable consists of roughly 6,300 lines of Java across 31 source files, a JSON-encoded world definition (data/game.json), and a set of location artwork. I led the high-level architecture, the core game-logic engine, and much of the GUI display logic.

The codebase is organized into a three-layer architecture: "domain / application / view":

  • domain/ - Pure model classes with no UI or I/O dependencies: World, Location, Connection, GameObject, Character, Inventory, the rule classes (UseRule, GiveRule), and the Selector hierarchy (NameSelector, AttributeSelector). These types describe *what* the game world is, independently of how it is loaded or displayed.
  • application/ - Use-case coordination: GameEngine (the central command processor), GameState (the player's current location, inventory, turn count, and message history), CommandResult (a structured outcome of a turn), WorldFactory (deserializes JSON into a populated World), and Launcher (the main entry point).
  • view/ - Swing UI components: MainWindow, LocationView, InventoryWindow/InventoryView, CommandPanel, MessagePanel, MenuView, TurnCounterView, WinDialog, and the GameController mediator that bridges UI events to engine calls.

This layering enforces a unidirectional dependency rule: view depends on application, application depends on domain, and domain depends on nothing project-specific. The benefit is that the domain model is independently unit-testable.

Data-Driven World Definition

The game world — locations, objects, characters, dialogues, connections, win conditions, and interaction rules — is described in an external JSON file rather than hard-coded. WorldFactory uses Google's Gson library to deserialize data/game.json into a tree of intermediate POJOs, then walks that tree to construct the domain model. Locations, objects, and characters are registered in lookup maps so that JSON references-by-name resolve cleanly to live object references at load time.

This data-driven approach means new puzzles, items, and entire areas can be added without recompiling: the JSON exposes fields such as requiresItem (gating travel), canPickUp, winOnPickup, gameEnding, attributes, useRules, giveRules, and per-character responses. This makes the content separate from the game engine, so that the game engine can be reused on arbitrary stories.

Rule System

Interactions between items are modeled by two declarative rule types:

  • UseRule fires when the player uses one object on another (e.g. *use Rope Coil on Broken Milking Machine*). It owns a subject selector, a target selector, narration text, and a payload of rewards: objects to grant, characters to introduce, and connections to unlock.
  • GiveRule fires when the player gives an object to a character (e.g. *give Map to Bob*) and similarly produces narration and rewards.

Rather than matching by string equality, both rules use the **Selector interface** with two concrete implementations: NameSelector (exact-name match) and AttributeSelector (tag-based match against attributes declared in the JSON). This is a small but well-applied use of the Strategy pattern: it lets a single rule apply to "any object tagged *fruit*" without enumerating individual items, which keeps the JSON authoring tractable as content grows.

GameEngine evaluates rules by iterating the rule list and asking each one whether it matches the current subject and target. The matching rule's narration becomes the turn's user-visible message, and its rewards are merged into the world state.

Command Processing and Game Loop

GameEngine exposes a verb-oriented API — move, pickUp, drop, examine, talk, use, give, buy — each returning a CommandResult carrying narration text, success flag, and a gameFinished indicator. After every successful action, the engine increments the turn counter via GameState, then runs checkEnd to detect turn-limit loss or win conditions (reaching the ending location, picking up a "win-on-pickup" object, or earning enough tickets). Tickets themselves are modeled as an ordinary GameObject accumulated in the player's Inventory, which keeps win-condition logic uniform with all other inventory queries.

The GameEngine is exposed as a singleton via getInstance() so that UI components which do not directly hold a reference can still query state — a pragmatic compromise rather than a textbook-pure design choice.

View Layer and Threading

The UI is built on Java Swing. MainWindow composes the location image (LocationView), the side CommandPanel of action buttons, the scrolling MessagePanel of narration, the TurnCounterView, and a MenuView. Inventory is shown in a separate InventoryWindow. The GameController is the sole class that knows about both the engine and the widgets: it listens for button clicks and popup-menu selections, translates them into engine commands, then re-paints the view from the returned GameState and CommandResult.

Launcher handles thread discipline carefully: world loading (file I/O and JSON parsing) runs on a background thread, while window construction and event handling are dispatched onto Swing's Event Dispatch Thread via SwingUtilities.invokeLater.
